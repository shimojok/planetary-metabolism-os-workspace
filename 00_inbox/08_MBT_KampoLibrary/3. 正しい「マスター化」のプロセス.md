この元のJSONファイルを `mbt_kampo_library_master.json` にリネームするだけでは**不十分です**。  
単にファイル名を変えただけでは、**中身（データ構造や項目名）** は何も変わらないからです。

---

## 📌 なぜリネームだけではダメなのか？

### ❌ リネームだけでは変わらないこと
- ファイルの中身（JSONの構造）はそのまま
- `"mbt_kampo_library"` というルートオブジェクトが残っている
- `"version": "2.44.0"` など、ファイル自体のバージョン情報が含まれている
- `"formulas"` の中身もそのまま

これでは「新しいマスターファイル」としては使えません。

---

## ✅ 正しい「マスター化」のプロセス

### 1️⃣ ファイル名を変更する（最初のステップ）
```bash
mv mbt_kampo_library_master_v2.44.json mbt_kampo_library_master.json
```
これでファイル名はマスターらしくなりますが、**中身はまだ不完全**です。

---

### 2️⃣ JSONの構造を整理する（最重要）
現在のJSONはこのような構造になっています：

```json
{
  "mbt_kampo_library": {
    "version": "2.44.0",
    "last_updated": "2026-02-25",
    "formulas": { ... }
  }
}
```

✅ 理想的なマスター形式（例）：
```json
{
  "version": "2.44.0",
  "last_updated": "2026-02-25",
  "total_formulas": 294,
  "formulas": { ... }
}
```

👉 `"mbt_kampo_library"` という1段階のネストをなくし、直下にメタ情報を置くのが一般的です。

---

### 3️⃣ 必要な項目を残し、不要なものを削除
元のファイルには以下のような「管理用」項目があります：
- `"total_formulas"` → ✅ 残す（294）
- `"fully_defined"` → ✅ 残す（191）
- `"classification_source"` → ✅ 残す
- `"mbt55_version"` → ✅ 残す

これらはマスターとして保持すべき情報です。

---

### 4️⃣ `"formulas"` の中身はそのまま活用
`"F001"` 〜 `"F228"` までの処方データは、そのままマスターの核として使います。  
ここは基本的に変更しません。

---

### 5️⃣ 保存形式の決定
最終的なJSONは以下のような形になります：

```json
{
  "version": "2.44.0",
  "last_updated": "2026-02-25",
  "total_formulas": 294,
  "fully_defined": 191,
  "classification_source": "Shimojo Phytochemical Classification",
  "mbt55_version": "MBT55-v2.44",
  "formulas": { ... }
}
```

## ✅ まとめ：やるべきこと
| ステップ | 内容 | 理由 |
|----------|------|------|
| 1️⃣ | ファイル名を変更 | 識別しやすくするため |
| 2️⃣ | JSONのルート構造を整理 | マスターとして扱いやすくするため |
| 3️⃣ | メタ情報を保持 | バージョン管理や統計に必要 |
| 4️⃣ | 処方データはそのまま保持 | データの中核だから |
| 5️⃣ | 新しいJSONとして保存 | システムで読み込めるようにするため |
