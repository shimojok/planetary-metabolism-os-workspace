結論は **「はい、JSON は単なるデータ形式なので、GitHub にデモシステムを構築する場合は “必ず” JSON を読み込むプログラミングコードが必要になる」** です。  
ただし、これは **正しいアーキテクチャの基本原則**であり、あなたの HealthBook／MBT55／AgriWare のような“データ駆動型エンジン”ではむしろ必須の設計です。

ここからは、なぜコードが必要なのか、どのようなコードが必要なのか、そしてあなたのプロジェクトにとってどんなメリットがあるのかを体系的に整理します。

## JSON が必要になる理由

### 1. JSON は「データの倉庫」であり、動かすには“エンジン”が必要

JSON は **データを記述するだけ**のフォーマットです。  
例：

```json
{
  "id": "F119",
  "name": "半夏瀉心湯",
  "mbt_pathway": "PATH_04"
}
```

これは「情報が書いてある紙」にすぎません。  
これを読み取って：

- MBT55 パスウェイにマッピング
- 疾病マトリックスと照合
- 推奨ファイトケミカルを抽出
- 代謝不全を解析
- レポートを生成

といった処理を行うには、**必ずプログラム（Python/JS）が必要**です。

## JSON を読み込むコードが必要な理由

### 1. データとロジックを分離するため

あなたの HealthBook コードは、現在：

- データ（方剤・疾病・問診）
- ロジック（解析・推奨生成）

が混在しています。

これを分離すると：

- データは JSON
- ロジックは Python

という **世界標準のアーキテクチャ**になります。

### 2. GitHub での再現性が高まる

Gates財団・Microsoft が最も重視するのは：

- **再現性**
- **透明性**
- **データとロジックの分離**
- **モデルの説明可能性**

JSON＋Python の構造はこれを満たします。

### 3. Azure 実装が容易になる

Azure Functions / Cosmos DB / Blob Storage は JSON をネイティブに扱います。  
つまり、**Azure に載せる前提なら JSON 化は必須**です。

## JSON を読み込むコードはどんなものか

あなたの HealthBook の構造に合わせると、必要なコードは次のようになります。

### ● 1. JSON ローダー

```python
import json

def load_json(path):
    with open(path, "r", encoding="utf-8") as f:
        return json.load(f)
```

### ● 2. JSON → データクラス変換

```python
def load_kampo_from_json(json_data):
    return [KampoFormula(**item) for item in json_data]
```

### ● 3. 統合エンジンに渡す

```python
kampo_db = load_kampo_from_json(load_json("data/MBT_Master_part01.json"))
engine = HealthBookIntegratedEngine(kampo_db=kampo_db)
```

### ● 4. 結果を Markdown に書き出す（Obsidian 用）

```python
def save_to_markdown(data, path):
    with open(path, "w", encoding="utf-8") as f:
        f.write("---\n")
        f.write(yaml.dump(data))
        f.write("---\n")
```

これらはすべて **ClaudeCode が自動生成できます**。
