# 1. 構造化とは何か（Vault と GitHub の両方で必要）

構造化とは、単にフォルダーを作ることではなく、

**① 情報の意味的まとまりを分ける  
② AI が参照しやすい単位に整理する  
③ GitHub のリポジトリ構造と対応させる  
④ LangGraph のノード構造と対応させる**

この 4 点を満たすこと。

つまり、構造化とは **「AI が理解しやすい知識の地図を作ること」**。

# 2. Obsidian の .md は分割すべきか？

結論：

**分割すべきノートと、分割すべきでないノートがある。**

### 分割すべきノート（AI が扱いやすくなる）
- 1 ノートに複数のテーマが混在している  
- 1 ノートが 10,000 文字以上  
- 理論・数式・コードが混ざっている  
- MBT55 の科学モデルと政策モデルが同じノートにある  
- AgriWare のデータ構造と農学理論が混ざっている  

AI は「1 ノート 1 トピック」の方が圧倒的に理解しやすい。

---

### 分割しなくてよいノート
- 1 つのテーマが一貫している  
- 1 ノートが 2,000〜5,000 文字程度  
- 章立てが明確で、内容が一貫している  
- そのノートが「ひとつの概念のまとまり」になっている  

AI は Markdown をそのまま読めるので、  
**無理に細かく分割しすぎる必要はない。**

---

# 3. Vault の構造化は必要か？

**必要。絶対に必要。**

理由は 3 つ：

### ① LangGraph のノードが「どの知識を参照するか」を明確にできる  
例：  
- MBT55 ノード → `/10_Knowledge/MBT55`  
- AgriWare ノード → `/10_Knowledge/AgriWare`  
- HealthBook ノード → `/10_Knowledge/HealthBook`  

### ② VS Code の AI エージェントが「どのフォルダーを読むか」を指定できる  
例：  
> このフォルダー `/20_Projects/MBT55_Agent` を参照してコードを書いて。

### ③ GitHub のリポジトリ構造と 1:1 で対応できる  
→ 投資家・財団に見せるときに圧倒的に理解しやすい。

---

# 4. 添付の Obsidian ノート（説明＋コード）は分割すべきか？

**内容によるが、基本は「説明」と「コード」は分けた方がよい。**

理由：

- AI は「説明」と「コード」を別々に参照した方が精度が高い  
- LangGraph のノードに渡すときも分けた方が扱いやすい  
- GitHub に載せるときも分けた方が見やすい  

### 最適な分割方法
```
MBT55_teleodynamics.md
MBT55_equations.md
MBT55_scientific_basis.md
MBT55_policy_impact.md
MBT55_agent_design.md
```

コードは `/40_Code/MBT55` に置く。
