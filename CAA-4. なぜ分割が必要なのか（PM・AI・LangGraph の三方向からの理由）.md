[[HB6. HealthBook Platform 実装工程計画書]]
このノートを何分割しますか？

---

結論から言うと、**そのノートは 1 つのままでは “AI にとっても人間にとっても扱いにくい巨大複合ノート” なので、確実に分割すべきです。**  
そして、あなたが貼ってくれた内容（HB6 実装工程計画書）は、**最低でも 6〜8 ノートに分割するのが最適**です。  
理由と、どう分割すべきかを PM 視点で整理します。

---

## 🔍 なぜ分割が必要なのか（PM・AI・LangGraph の三方向からの理由）

### 1. **PM 視点：プロジェクト管理上、1 ノートに複数領域が混在すると破綻する**

HB6 のノートには以下が全部混ざっています：

- プロジェクト計画
- 工程表
- 技術スタック
- リスク管理
- チーム構成
- 成果物一覧
- マイルストーン
- QA 戦略
- インフラ構成
- API 設計
- コードの説明

これは PMBOK でいう「WBS（Work Breakdown Structure）」が未分離の状態。  
**管理不能になる典型パターン**です。

---

### 2. **AI 視点：1 ノートに複数テーマがあると、AI が誤読・混乱する**

VS Code の AI や LangGraph のノードは、  
**1 ノード = 1 機能 / 1 役割 / 1 知識領域**  
で動くのが最も精度が高い。

1 ノートに複数領域が混ざると：

- AI がどの部分を参照すべきか判断できない
- コード生成の精度が落ちる
- LangGraph のノード設計が複雑化する
- RAG の検索精度が落ちる

つまり、**AI の性能が 30〜50% 落ちる**。

---

### 3. **LangGraph 視点：ノードとノートは 1:1 が最適**

LangGraph のノードは：

- 問診解析ノード
- 疾病マトリックスノード
- ファイトケミカルノード
- MBT55 代謝ノード
- 健診データ統合ノード
- API ノード
- Azure デプロイノード

など、**機能単位で分かれる**。

1 ノートに全部入っていると、  
**LangGraph のノード設計が不可能になる**。

---
