## AI 自動分割のメリット（Kaz のプロジェクトに最適な理由）

### ● 1. マルチエージェントが理解しやすい粒度を AI が判断できる  
エージェントごとに必要な情報は異なるため、  
- MBT55 理論エージェント  
- AgriWare 作物モデルエージェント  
- HealthBook 吸収モデルエージェント  
- AGRIX 政策モデルエージェント  

などが **自分に最適な粒度でノートを参照できるようになる**。

### ● 2. LangGraph のノード構造と自然に一致する  
LangGraph は「1 ノード = 1 機能 / 1 思考ステップ」が基本。  
AI が自動で分割すると、  
**ノードとノートが 1:1 で対応しやすくなる**。

### ● 3. Vault の巨大ノートを AI が自動で整理してくれる  
Kaz の Vault は膨大で複雑。  
AI 自動分割は、**知識の地図を自動生成する**ことになる。

### ● 4. GitHub のリポジトリ構造も自動生成しやすくなる  
分割されたノートはそのまま  
- `/10_Knowledge`  
- `/20_Projects`  
- `/30_AI_Agents`  
- `/40_Code`  
に配置できる。

## AI 自動分割の注意点（問題ではなく“前提条件”）

### ● 1. AI が誤って「分割しすぎる」可能性  
AI は「粒度を細かくしすぎる」傾向があります。  
例：  
- 1 ノートが 20 個に分割される  
- 1 段落ごとに別ノートになる  
- 似た内容が複数ノートに分散する  

→ **対策：分割定義書（JSON）に“最大分割数”を設定する**

```json
{
  "max_splits": 12,
  "min_chunk_size": 500
}
```

---

### ● 2. AI が「文脈をまたいで分割」する可能性  
例：  
- 工程計画の途中で分割  
- 技術スタックの説明が途中で切れる  

→ **対策：AI に「意味単位で分割する」ルールを与える**

---

### ● 3. AI が「誤ったフォルダーに保存」する可能性  
例：  
- MBT55 のノートが HealthBook に入る  
- 技術スタックが AGRIX に入る  

→ **対策：フォルダー分類ルールを JSON に含める**

```json
{
  "folder_rules": {
    "MBT55": "/10_Knowledge/MBT55",
    "HealthBook": "/10_Knowledge/HealthBook",
    "AgriWare": "/10_Knowledge/AgriWare",
    "AGRIX": "/10_Knowledge/AGRIX",
    "Tech": "/10_Knowledge/Systems_Architecture"
  }
}
```

---

### ● 4. AI が「重要な文脈を失う」可能性  
例：  
- リスク管理と工程計画が分離されてしまう  
- 技術スタックと API 設計が切り離される  

→ **対策：AI に“文脈保持”のルールを与える**

```json
{
  "keep_context": true,
  "context_window": 2
}
```

## AI 自動分割を安全に実行するための設計（最適解）

### ● 1. 分割定義書（JSON）  
AI が自由に分割するが、  
- 最大分割数  
- 最小チャンクサイズ  
- フォルダー分類ルール  
- 文脈保持ルール  
を JSON で制御する。

### ● 2. LangGraph の 5 ノード構成  
1. **LoadMarkdown**  
2. **SplitPlanner（LLM）**  
3. **ChunkValidator（LLM）**  
4. **FileWriter（Python）**  
5. **Verifier（LLM）**  

→ 自動分割の安全性が高まる。

### ● 3. GitHub への自動コミット  
分割後のノートを GitHub に自動 push する。
