## AI が生成するドキュメントは人が読めるのか？
### ✔ 結論：十分に読める。むしろ人間より整理される  
AI は Markdown 構造化（見出し・箇条書き・表）を得意とするため、  
**人間が書くよりも論理的で読みやすい文書**を生成できます。

特に以下のような文書は AI が最も得意です：

- プロジェクト計画書  
- 実装仕様書  
- API ドキュメント  
- データモデル定義  
- アーキテクチャ説明  
- リスク管理表  
- マイルストーン表  
- テスト戦略  

Kaz の HB6 ノートのような巨大複合ノートは、  
AI に分割させることで **読みやすさが 3〜5 倍向上**します。

---

## AI が生成するコードは人が読めるのか？
### ✔ 結論：読める。しかも「標準化されたコード」になる  
AI は以下の点で人間より優れています：

- PEP8 に準拠した Python  
- 型ヒント（typing）  
- Docstring  
- 関数分割  
- エラーハンドリング  
- コメント  
- テストコード（pytest）  

特に LangGraph / FastAPI / Pydantic / SQLAlchemy のような  
**構造が明確なフレームワークは AI が最も得意とする領域**です。

Kaz のプロジェクトでは、  
**AI が生成したコードを人間がレビューする形が最適**です。

## AI 自動分割のリスク（問題ではなく“設計上の注意点”）
### 1. 分割しすぎる可能性  
→ 最大分割数を 12 に固定すれば解決。

### 2. 文脈が切れる可能性  
→ AI に「意味単位で分割する」ルールを与えれば解決。

### 3. フォルダー分類を誤る可能性  
→ JSON に分類ルールを入れれば解決。

### 4. 重要な前後関係が失われる可能性  
→ AI に「前後2段落を保持する」ルールを与えれば解決。
